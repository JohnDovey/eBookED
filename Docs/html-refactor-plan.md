# eBook Editor — Markdown → HTML Content Model Refactor

## Context

eBook Editor currently stores every chapter/front-matter/back-matter page as Markdown (YAML front matter + Markdig-flavored Markdown body), and each export format (EPUB, PDF, Word) either converts that Markdown to HTML first (EPUB) or walks the Markdig AST directly (PDF, Word) with duplicated block/inline-handling logic. The Preview window uses a *third*, entirely independent Markdown parser (Markdown.Avalonia) that has no support for this app's custom-container/attribute-block syntax at all, requiring a regex sanitizer just to avoid rendering visibly broken text. This session also surfaced a real, confirmed Markdig bug (a trailing `{.class}` attribute block immediately after a pipe table breaks the table parser entirely) that had to be worked around with nested custom containers instead of a literal Markdown table for the Insert Image feature — a class of parsing landmine that cannot recur with real HTML markup.

The user wants to flip the whole content model to HTML: store chapters as HTML, import converts to HTML, and the editor becomes an HTML editor with three modes — raw HTML text editing, a WYSIWYG rich-text mode, and the existing standalone Preview window. They also want to eliminate the app's one vendored (non-NuGet) dependency (AvaloniaEdit, vendored solely because of a Markdown.Avalonia-caused assembly collision) and rely on NuGet for everything else, and asked for an honest feasibility assessment rather than a plan that just agrees.

**Feasibility verdict**: the storage-format flip is a sound *engineering* decision — it consolidates three different parsers (Markdig for EPUB/PDF/Word, Markdown.Avalonia for Preview) onto one real HTML DOM (via AngleSharp), and categorically eliminates the confirmed table/attribute-block parsing bug. It also does eliminate the vendored dependency, confirmed by reading `AvaloniaEdit`'s own vendoring comment: the collision is specifically `Markdown.Avalonia.SyntaxHigh` reflection-loading the real `Avalonia.AvaloniaEdit` NuGet assembly, which disappears once `Markdown.Avalonia` is removed. But raw HTML is *not* an easier hand-authoring format than Markdown — it's more verbose for exactly the constructs this app's authors touch most (footnotes, tables, emphasis, links). "Easier editing" is only true if the WYSIWYG mode is genuinely good, which is real, open-ended engineering work, not a free side effect of the storage change. The user confirmed (via the three decisions below) that this work should be fully funded rather than shipped as a downgrade.

**Decisions locked in with the user**:
1. **WYSIWYG via `Avalonia.Controls.WebView`** (now MIT/free/official as of Avalonia 11.4.0, Oct 2025 — not the Electron-style embed the original "no WebView" decision was reacting to; uses the OS's native webview, no bundled Chromium). The Preview window migrates to the same tech in the same effort envelope, giving Preview a real CSS engine for the first time.
2. **Cut the "Markdown (Whole Book/Chapter)" export menu feature.** No HTML→Markdown converter exists today (only the reverse direction, which is itself being deleted); building one is out of scope. `ReverseMarkdown` and `MarkdownExportService` are deleted cleanly, no compromise needed.
3. **Fund real CSS-in-PDF/Word now**, via `AngleSharp.Css`'s cascade/specificity resolution, as in-scope work in the PDF/Word rewrite phase — finally closing the long-documented "Apply Style/Caption render as plain unstyled text in PDF/Word" limitation.

## New content-format design

Keep the existing `---\nYAML\n---\n\n<body>` file convention (`ChapterFileService`'s parse/write logic and `ChapterFrontMatter`'s shape are already format-agnostic) — only `<body>` changes from Markdown to an HTML *fragment* (no `<html>/<head>/<body>` wrapper; each exporter already wraps the body into its own document shape). Use a distinct file extension (recommend `.ebhtml`) rather than silently keeping `.md` — the body is a fragment, not standalone valid HTML/XML, and treating it as interchangeable with plain `.md` would be misleading. Thread this through `ChapterFileNaming`, `CreateNewChapterFile`, `SyncChapterFileNames`, and `OrphanChapterScanner`. Bump `ProjectFile.SchemaVersion` (already exists, currently `1`, in `eBookEditor.Core/Models/ProjectFile.cs`) so legacy `.md` projects are recognized and gated behind the migration tool (Phase 6) rather than silently mixed with new-format projects.

- `PageGeneratorService`/`BookIndexGenerator` (`eBookEditor.Markdown`) rewritten to emit HTML strings instead of Markdown strings — note this introduces a real new bug surface (HTML-encoding interpolated titles/author names) that the Markdown generator never needed. Relocate both (and any other surviving format-neutral generation logic) into a new small project with no Markdig dependency, since `eBookEditor.Markdown` is retired at the end (Phase 9).
- `EditorStyleCatalog`'s 11 class names are unchanged; only the wrapper syntax changes from `::: {.class} ... :::` to real HTML — `<span class="...">` for inline styles (Small Caps, Underline, Strikethrough, Monospace, Sans-serif, All Caps), `<div class="...">` for block styles (Verse, Inset, Attribution, Drop Cap, Caption).
- `InsertTableWindow` builds a real `<table><thead>...<tbody>...</table>` instead of a GFM pipe-table string — this is what categorically retires the Markdig table/attribute-block bug.
- `OnInsertImageClick`'s nested-custom-container workaround becomes real `<figure><img src="..." alt="..."><figcaption class="caption">...</figcaption></figure>` — simpler and more semantically correct than the Markdown workaround it replaces.
- No toolbar command exists today for footnotes (confirmed no `OnInsertFootnote`-style handler exists) — since HTML has no terse footnote idiom like `[^1]`, add an "Insert Footnote" command as new, explicitly-funded scope so footnote-authoring ergonomics don't quietly regress.

## Versioning and commit cadence

- **Start of this initiative**: bump the major version (currently `1.11.0` → `2.0.0`, per this repo's existing bump-policy convention of major/minor/patch reflecting change scope) in the same commit as this plan being adopted, then commit and push immediately — this marks the refactor's starting point in history before any implementation code changes.
- **After each phase below** (0 through 9): bump the minor version (`2.1.0` after Phase 0, `2.2.0` after Phase 1, … `2.10.0` after Phase 9), commit, and push — only once that phase's own build/test/verification criteria (see Verification, below) are green. Each phase is its own reviewable, revertible commit; don't batch multiple phases into one version bump.
- Follow this repo's standing practice for every commit: `dotnet build` (0 warnings/errors) and `dotnet test` (full suite green) before bumping/committing, a version-bump comment in `Directory.Build.props` explaining what the phase did (matching the existing changelog-comment convention already in that file), and push straight to `main` without asking (per this repo's established preference) once a phase's own verification passes.

## Phased plan

**Phase 0 — WebView spike (de-risking, before any production code). ✅ Done — result: PASS, WYSIWYG approach confirmed viable.** Built a standalone `Avalonia.Controls.WebView` (`NativeWebView`, on macOS backed by WKWebView) proof of concept with a real, automated round-trip test (no manual interaction needed, since this sandbox can't drive real mouse/keyboard input): `InvokeScript` sets up a `contenteditable` div and an `input` listener, then programmatically edits the DOM to `<p>Edited via script.</p><figure><img src="x.png"><figcaption class="caption">A caption</figcaption></figure>` (the exact Insert-Image markup shape from this plan) and dispatches an `input` event; the listener calls the JS-global `invokeCSharpAction(...)` (auto-injected by `NativeWebView`, no manual bridge setup needed); C# receives it via `WebMessageReceived` and confirmed the round-tripped HTML matched exactly. `document.execCommand` was also confirmed present (informational only — the plan still recommends explicit JS DOM-manipulation functions over `execCommand` for production, per §8's reasoning).
  - **Real, reproducible bug found and fixed**: constructing/attaching `NativeWebView` to a `Window` before that window has completed a real Avalonia layout pass crashes the app natively — `EXC_BAD_INSTRUCTION` inside `-[WKWebView initWithFrame:configuration:]` → `_NSViewValidateGeometry`, reported as `"Invalid view geometry: y is NaN"`. Reproduced twice, independent of this sandbox's known idle-display quirk (tried `caffeinate` first; didn't help). **Fix, confirmed working**: defer setting `Window.Content = webView` (and the first `NavigateToString`/`Navigate` call) until after the window has shown and completed layout — e.g. `Dispatcher.UIThread.Post(() => { window.Content = webView; webView.NavigateToString(...); }, DispatcherPriority.Loaded)` rather than assigning `Content` eagerly at construction time. **Phase 8 must apply this exact pattern** when attaching `NativeWebView` in both the main editor window and `PreviewWindow` — don't rediscover it the hard way.
  - Scratch spike lives at `/private/tmp/claude-501/-Volumes-JohnDovey-Projects-eBookED/ae1d51a3-3e87-4ed4-a5db-993c2b385743/scratchpad/webview-spike/` (outside the repo, per the plan's "standalone, outside the main app tree" instruction) — not committed, safe to discard; kept only as a working reference for Phase 8.

**Phase 1 — Core storage format change.** New `.ebhtml` extension threaded through `ChapterFileNaming`/`ChapterFileService`/`CreateNewChapterFile`/`SyncChapterFileNames`/`OrphanChapterScanner`; bump `ProjectFile.SchemaVersion`. Downstream generators/exporters are not touched yet and will mechanically break on HTML input — expected, resolved by Phases 2–5. Update `ChapterFileServiceTests`/`ChapterFileNamingTests`. Lowest-risk phase.

**Phase 2 — Content-generation rewrite.** `PageGeneratorService`/`BookIndexGenerator` emit HTML per the design above, relocated out of `eBookEditor.Markdown`. Rewrite their tests for HTML shape and HTML-encoding correctness.

**Phase 3 — Import pipeline rewrite.** `OpenXmlToMarkdownConverter` → `OpenXmlToHtmlConverter` (same DOM-walking approach, HTML emission target). Delete `HtmlToMarkdownConverter`/`ReverseMarkdown` entirely (confirmed unnecessary once HTML import is a parse/sanitize/pass-through step via AngleSharp rather than a conversion — and confirmed safe to fully drop given the Markdown-export cut). Update `ChapterImportService`'s `.md`/`.docx`/`.html` routing (`.md` → Phase 6 migration path, `.docx` → new converter, `.html` → simplified AngleSharp pass-through).

**Phase 4 — EPUB export simplification.** `EpubBuilder` drops its `MarkdownToHtmlConverter` step entirely — the stored HTML body wraps directly into the XHTML shell. Smallest, most mechanical export-side phase; sequence right after Phase 3 so there's a fully-working, demoable export path while Phase 5 is still in progress.

**Phase 5 — PDF/Word export rewrite (highest-effort phase, now includes real CSS).** `MarkdownToPdfRenderer` → `HtmlToPdfRenderer`, `MarkdownToDocxConverter` → `HtmlToDocxConverter`, both rewritten to walk an AngleSharp-parsed DOM instead of a Markdig AST, preserving the existing "switch over node type → emit QuestPDF/OpenXml calls" structure — do **not** touch the pagination logic itself (roman/arabic numbering, verso/recto running headers via `.Section()`/`.CaptureContentPosition()`, clickable TOC via `SectionLink`) in this phase, only its input source, to keep regression risk isolated. Add `AngleSharp.Css`-driven cascade resolution so `EditorStyleCatalog` classes (and any other CSS the selected template defines) actually render in PDF/Word, not just structurally survive unstyled — real, meaningful scope added to an already-large phase, budget accordingly. Evaluate `HtmlToOpenXml` (NuGet, AngleSharp+OpenXml-based) as a foundation for the Word side; expect to still need custom code on top for real `FootnotesPart` footnotes, exact EMU image sizing, and the thematic-break-as-page-break convention. No PDF-side shortcut exists — `HtmlToPdfRenderer` is a from-scratch rewrite of comparable scope to today's `MarkdownToPdfRenderer`.

**Phase 6 — Legacy content migration tooling.** Keep Markdig as a NuGet-only, migration-only dependency — reusing `MarkdownToHtmlConverter`'s existing Markdig wrapper for a one-time batch conversion is better engineering than hand-rolling parsing for a rarely-run tool, and has no relationship to the vendoring problem. Ship as an explicit, user-triggered "Upgrade Project to HTML…" command (not silent auto-migration), gated on `ProjectFile.SchemaVersion`, with a backup/copy prompt before mutating a real project in place and an idempotency check against double-conversion. Treat any hand-edited generated pages (title/imprint/TOC/about-author) like ordinary chapter files during migration — convert the body verbatim, don't regenerate from metadata, so hand-edits aren't silently discarded. Acceptance criteria: run the migration against `manual/` (the real, actively-maintained project checked into this repo) and against `/Volumes/JohnDovey/Documents/Test Book`, then re-export EPUB/PDF/Word from each and diff against pre-migration exports. Can run in parallel with Phase 5 — depends only on Phases 1–3.

**Phase 7 — Editor UI raw-mode swap.** Swap the vendored `AvaloniaEdit` project reference for the NuGet `Avalonia.AvaloniaEdit` package (only completes once Phase 8 removes `Markdown.Avalonia` — sequence 7 and 8 as one combined milestone). Swap the AvaloniaEdit syntax-highlighting definition from Markdown to HTML. Update `InsertTableWindow`/`OnInsertImageClick`/`OnApplyStyleClick` to emit HTML, and add the new "Insert Footnote" command.

**Phase 8 — WYSIWYG build-out (highest risk).** Add `Avalonia.Controls.WebView`; build a shared embedded HTML/CSS/JS editor-shell asset with the project's actual selected CSS template linked in; wire `InvokeScript` (C#→JS content push) and `WebMessageReceived` (JS→C# edited-content pull, debounced); implement toolbar commands (Insert Table/Image/Footnote, Apply Style) as explicit JS DOM-manipulation functions, not `document.execCommand` (avoids betting on legacy/deprecated browser behavior). Migrate `PreviewWindow` to the same WebView tech, `contenteditable="false"`, no change-listener — this is what lets Phase 7's NuGet swap fully complete, and retires `PreviewMarkdownSanitizer` for good. Raw mode and WYSIWYG mode share one HTML-building layer (e.g. one function returning a `<table>...</table>` string) consumed two ways: string-inserted at the AvaloniaEdit caret vs. DOM-inserted via the JS layer — don't force a single generic implementation across both insertion mechanics. Timebox this phase; Phase 0's spike results are the go/no-go signal for full scope.

**Phase 9 — Cleanup.** Delete `src/AvaloniaEdit/` entirely. Retire `eBookEditor.Markdown` (should be empty after Phases 2–4 except the Phase-6 migration code, which moves to a small dedicated project, e.g. `eBookEditor.Migration`). Confirm `Markdown.Avalonia`/`ReverseMarkdown` are fully removed from every `.csproj`.

**Risk ranking**: Phase 8 (WYSIWYG) highest, Phase 5 (PDF/Word + real CSS) second, everything else materially lower.

## Test-suite impact

Confirmed real, not hand-waved: at least 16 of the ~29 files under `tests/eBookEditor.Tests/` reference Markdown/Markdig/`.md` directly by name, and that's a floor — files like `EpubBuilderTests`/`PdfBuilderTests` embed literal Markdown fixture strings without the word "Markdown" appearing in the test file itself. Expect roughly half to two-thirds of the suite to need direct rewrites (fixture strings changed from Markdown to HTML) plus new test files for every net-new converter (`OpenXmlToHtmlConverterTests`, `HtmlToPdfRendererTests`, `HtmlToDocxConverterTests`, migration-tool tests). This is comparable in size to the production-code rewrite, not an afterthought — each phase above should carry its own test rewrite as part of that phase's completion criteria, matching how this app's own build history has always worked, rather than deferring test migration to one giant end-of-project pass.
